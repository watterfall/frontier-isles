# 桥接挑战 v1：让一次类比对应可以被支持或打断

日期：2026-07-19
状态：实现契约（承接《连接档案 v1》§5 的"等各自的有效挑战合同成熟后再开放写入"）

## 1. 这一轮解决什么

连接档案 v1 只对证据与反证路径开放回应；桥接路径（类比映射、假说形式化、概念原型、设计虚构）只读。但档案契约的第 4 问——"什么测试能区分支持、反驳或**仅仅是表面相似**"——恰恰以桥为最主要对象：一座桥就是一条被作者化的对应论断。本轮把既有回应契约原样推广到桥接路径，不发明新表单、不新增动词。

## 2. 语义与唯一事实来源

- 一件桥接工件是内容寻址、有作者、锚定在账本上的**对应论断**，与 claim 同构地可被挑战。
- 回应仍是 `validate` / `refute`，目标是**桥接工件的 ref**。锚事件可以是 `bridge_artifact`（MCP/网关路径）或携带该工件的人工 `transplant`（B.3 渡口路径）。
- 回应投影为一条**关于这座桥的有符号 evidence 流**（记录保留桥 ref 与回应工件）；桥流本身保持中立，`proposed → ratified` 的成熟度语义只由 `bridge_accept` 改变——**任何数量的 validate 都不会自动批准一座桥**。
- 回应岛必须不同于桥的锚定岛（不允许自我背书）；证据描述、正文、区分性测试的必填与校验规则与档案 v1 完全一致。
- 机制汇聚与谱系路径本轮继续只读，各自的挑战合同（结构改写/边界细化、谱系争议）另行成熟后再开放。

## 3. 界面

- 桥接路径的档案下方出现与证据路径相同的渐进式回应表单，立场用具体语言：「这次对应经得起检验」/「这次对应在此断裂」，不暴露内部类名。
- 桥的档案联查兄弟路径：凡记录目标指向本桥 ref 的有符号回应，以「关于这次对应的跨岛回应」列出，可一键聚焦到该回应路径细读全档。
- 成功写入后仍沿档案 v1 规则刷新连接场并聚焦新记录（新记录出现在对应的 evidence/contradiction 路径中）。
- 移动端保持读取对等、不开写入口。

## 4. 非目标

- 不因支持性回应自动批准（ratify）桥；批准仍是 `bridge_accept` 的人为决定。
- 不开放机制/谱系写入。
- 不把"一次回应"表述为"两个领域等价已被证明"。

## 5. 验收

- 服务端：对 `bridge_artifact` / `transplant` 锚定的 ref 回应成功（201，事务完整）；锚定岛自我回应被 `same_island` 拒绝；无锚 ref 仍拒绝。
- 核心：语义回应加入关于桥的有符号流且保留桥 ref；桥流中立性与成熟度不受污染（测试钉住）。
- 桌面端：桥路径出现具体立场表单；兄弟回应联查可聚焦；数学/谱系路径无表单。
- 全量测试、类型检查、构建与 e2e 通过。
