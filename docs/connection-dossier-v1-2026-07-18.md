# 连接档案 v1：让跨领域关系可以被读懂、检验与改写

日期：2026-07-18
状态：实现契约

## 1. 这一轮解决什么

连接场已经让用户看见不同问题之间的数学、机制、证据、反证、桥接与谱系关系，但“看见一条线”还不是“打通内在联系”。真正可用的连接必须回答：

1. 原问题到底提出了什么论断？
2. 另一个领域以什么材料支持或反对它？
3. 两边相同的是哪一个可迁移机制，差异和适用边界在哪里？
4. 下一步做什么测试，能够区分支持、反驳或仅仅是表面相似？
5. 谁在何时作出了回应，原始材料还能否被追溯？

因此 v1 不再增加另一张图，而是在被选中的连接后展开一份“连接档案”，并允许具备权限的人从回应岛屿写入一次可追溯的支持或质疑。

## 2. 唯一事实来源

- 连接仍由账本事件投影产生，不建立第二套关系数据库。
- 原论断、发布物和回应正文仍是内容寻址 ref；界面只呈现解析后的摘要，并保留原 ref。
- 支持与反证仍分别使用 `validate` 和 `refute`，不能合并成一个模糊的“相关”。
- 有效回应沿既有规则落入 Workshop（方法与试验）；权限、目标或证据校验失败时事务不写入任何 ref、事件或落位。
- 离线 fixture 只保留真实拓扑与有限元数据；无法解析的正文必须明确显示为不可用，不生成替代内容。

## 3. 投影数据契约

每条账本连接可以包含多条 `records`，因为同一对岛屿、同一种关系可能由多个目标论断和多次回应共同形成。

```ts
interface CurrentRecord {
  targetRef: string;
  targetKind?: string;
  targetSummary?: string;
  targetEvidence?: EvidenceRef;

  responseRef?: string;
  responseKind?: string;
  responseBody?: string;
  responseTest?: string;
  responseEvidence?: EvidenceRef;

  action: ActionType;
  actor: string;
  actorKind: ActorKind;
  ts: string;
  historical: boolean;
}
```

投影规则：

- 旧事件把 `event.ref` 直接指向目标论断，继续按旧方式工作，并标记 `historical: true`。
- 新的 `validate` / `refute` 事件把 `event.ref` 指向回应产物；回应正文中的 `targetRef`（兼容 `ref`）才是语义目标。
- 投影必须同时保留 `targetRef` 和 `responseRef`，不能为了聚合而丢弃回应正文、证据或作者信息。
- 摘要只读取明确的字符串字段：目标的 `text` / `title` / `body`，回应的 `body` 和 `test`。未知对象不得直接序列化成“正文”。
- 没有 resolver 时仍产出基本记录，正文和证据保持缺省。
- 记录按目标 ref、时间、回应 ref 稳定排序；现有连接的 `weight`、方向、符号和成熟度语义保持不变。

## 4. 回应写入契约

桌面端仅对证据与反证连接开放回应。用户从路径的回应端岛屿，以明确身份写入：

- 立场：支持（`validate`）或质疑（`refute`）；
- 回应正文：另一个领域的材料究竟说明了什么；
- 区分性测试：什么结果会支持、削弱或推翻这次对应；
- 证据描述：可追溯的 RO-Crate 地址与内容哈希。

服务端以一个事务完成：

1. 验证目标 ref 确实由另一座岛的 claim/publish 事件锚定；
2. 验证回应岛、动作、正文、测试与证据描述；
3. 复用既有能力检查与人工/代理降级规则；
4. 把回应内容保存为独立 ref；
5. 账本事件指向回应 ref，回应内容保留 `targetRef`；
6. 返回目标 ref、回应 ref、有效动作和落点。

服务端不替用户选择回应岛，也不把无证据评论伪装成支持或反证。权限不足、证据不完整和目标无效都必须成为可见错误。

v1 只验证证据描述的必填字段、角色和 sha256 形状，不联网检查 RO-Crate 是否可达，也不重新计算远端内容哈希；界面和文档不得把“已登记证据描述”写成“证据内容已验证”。

## 5. 桌面交互

选中连接后，原来的通用说明替换为一张展开的研究纸页：

1. **原论断**：可读摘要、来源岛屿、目标 ref、原始证据；
2. **跨域回应**：支持或质疑正文、回应岛屿、作者与时间、回应 ref、回应证据；
3. **区分性测试**：明确列出下一步能改变判断的观察或实验；
4. **缺口**：历史事件没有正文/测试时直接说明缺失；
5. **作出回应**：在证据/反证路径中渐进展开，成功后刷新同一连接场并聚焦新记录。

数学、机制、桥接与谱系路径在 v1 只读。它们继续显示公式、共享机制、边界和来源，等各自的有效挑战合同成熟后再开放写入。

视觉上保持“温暖纸张上的活体研究图谱”：档案像从选中路径展开的批注证据页，而不是仪表盘卡片、聊天框或独立模态窗。

## 6. 移动端

- 移动端显示与桌面端相同的原论断、回应、证据、测试、作者、时间和缺失状态。
- v1 保持只读，不在狭窄视口塞入证据写入表单。
- 所有 ref 和长地址必须可换行，不产生水平溢出。

## 7. 非目标

- 不自动推断未入账的跨领域关系。
- 不用大模型生成“相似性解释”填补资料空缺。
- 不把所有路径强行统一成一种挑战表单。
- 不在本轮引入通用评论系统、聊天系统或新图数据库。
- 不声称一次回应已经证明两个领域等价。

## 8. 验收条件

- 核心测试证明新式回应能按 `targetRef` 加入原连接，同时保留回应 ref、正文、测试、证据和来源。
- 旧式直接 ref 事件继续产生相同的连接与漩涡，并明确显示历史资料缺口。
- API 测试覆盖成功写入、权限不足、证据缺失、无效目标和跨岛约束。
- 桌面端能阅读完整档案并提交一次真实回应；刷新后记录仍在。
- 移动端能读取相同档案但没有写入口。
- 中英文、键盘焦点、窄屏换行和 reduced-motion 均通过检查。
- 全量测试、类型检查、构建和真实浏览器验证通过；已有 bundle 体积警告单独记录，不伪装成本轮回归。
